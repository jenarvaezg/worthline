# Historical depth is ingested by the assistant, not entered in the wizard

The onboarding **wizard captures present state** only — "what you have today" — creating opening facts dated today. Reconstructing an asset's _history_ (dated operations, valuation anchors, amortization schedules) is deliberately NOT a wizard job: filling years of dated facts through forms is unusable, and most of that history already exists as documents (broker exports, Excel ledgers, amortization plans). Historical depth is therefore an **optional, progressive, per-asset deepening** done through the **financial assistant**, after or alongside onboarding — never a precondition for a complete-enough present-state picture. The wizard stays lean; history lives outside it.

This extends ADR 0057 (assistant proposals: preview → confirm) to bulk, file-driven, historical ingestion. The assistant's only jobs are the two ends the LLM is good at: **extraction** (turn an uploaded, unstructured document into a proposal of typed facts) and **narration** (render the preview). It targets the **existing command vocabulary** — operations, valuation anchors, amortization plans, balance rebaselines, rate revisions — not a parallel write API, and every write goes through the same deterministic seam, validation, ripple, audit trail, and reversibility as manual entry. Hard rule: **the LLM never computes money.** Every figure in the preview — position impact, reconciliation, anomaly flags, the resulting curve — is computed by the deterministic engine; the assistant only presents them. The server action remains the trust boundary (ADR 0057): it re-parses, re-validates, stamps provenance, and writes through the manual-entry seam.

Because a bulk historical proposal (hundreds of dated facts) cannot be reviewed row by row, trust rests on two things the user _can_ judge — not on trusting the model:

1. A **reconciliation anchor** — the engine confirms the reconstructed facts reproduce a present-day position the user already knows (typically the wizard's present-state declaration, or the document's own closing balance). This is the checksum that would have caught the €186k phantom position from the July 2026 statement re-import.
2. An **engine-computed reviewable summary** — the resulting value curve, per-holding aggregates, prose, and flags for the few anomalies (oversell, gaps, sign flips, same-day collapses). The user sanity-checks the _shape_, not the rows.

Endpoint reconciliation plus shape review is necessary but not sufficient: an extraction error that preserves the endpoint (swapped dates, a buy/sell pair that still nets) can pass undetected. So assistant-ingested facts are stamped `source: agent` and treated as a distinct **provenance tier** — "reconstructed, reconciled to today" — on a spectrum below broker-verified and hand-entered facts. The assistant is not restricted to structured documents; an arbitrary Excel is allowed, but its output carries the reconstructed tier while a broker export or amortization plan carries a higher one. The app is transparent about which history is reconstructed and lets the user correct individual points later. Fine-grained historical trust comes from _knowing what is reconstructed and being able to fix it_, not from assuming the model was right.

## Consequences

- **Persisted proposal, but only the structured extraction.** Bulk ingestion spans multiple files and turns, so the proposal is persisted (survives turns, accumulates files, confirmable later) — unlike the ephemeral single-tool preview of ADR 0057. What is persisted is the **extracted typed commands + a document reference (name/hash) + provenance**, in the workspace's existing Turso DB — **never the raw document**, which lives only in-memory during the extraction request. No blob store (no S3). The LLM runs once (extraction); the preview is a deterministic, cheap projection over the stored commands; matching and reconciliation are re-derived against live data **at confirm**, never trusting the stored preview (as the statement importer already does).
- **Extraction is classify-then-extract, at the segment level.** A mixed "junk-drawer" document is segmented and grouped by fact kind / asset type to pick the per-type schema; matching and reconciliation are per individual holding. Conversational turns are the disambiguation mechanism when classification or a column is ambiguous — not narration for its own sake.
- **The reconstructed tier splits reconciled vs unverified** (see CONTEXT.md): items with a reconciliation anchor whose checksum passes vs items with no anchor (a holding no longer held, present only in the document), which the preview forces the user to eyeball.
- **The confirm applies all-or-nothing with a single batched ripple.** A bulk historical apply (hundreds of dated facts) must never ripple per-fact — that is the O(N×snapshots) cliff that produced 504s and minute-long operations this session (see the ripple-perf issue). Today's batched single-pass ripple is investment-only; a mixed apply needs it **extended to debt and housing facts** before junk-drawer ingestion is allowed. The pilot is therefore **one domain per confirm** (the investment statement reuses the existing batched apply); multi-domain batched apply is built when the junk-drawer case arrives.
- Build order: investment statement (pilot, reuses the existing pipeline) → amortization schedule (as a dated balance series, ADR 0056) → real-estate valuation anchor. The junk-drawer is not a separate milestone — it is the classifier routing into these typed extractors once they exist.
- The preview must show **position impact (before → after)**, not operation counts — the gap tracked for the statement importer (see the import-correctness epic).
- Reconciliation anchors give onboarding's present-state declaration a second job beyond seeding today's figure; it becomes the checksum for later historical ingestion.
